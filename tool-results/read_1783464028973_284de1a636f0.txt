   194тЖТ## Phase 3: The Loop (1 script)
   195тЖТ
   196тЖТ### The Ralph Loop Wrapper
   197тЖТ
   198тЖТFile: `~/.config/opencode/webforge-loop.sh`
   199тЖТ
   200тЖТ**What it does:** Keeps Hermes running autonomously until the project is done.
   201тЖТ
   202тЖТ```bash
   203тЖТ#!/bin/bash
   204тЖТ# WebForge Autonomous Loop (Ralph Loop pattern)
   205тЖТ# Keeps re-launching Hermes until the plan is 100% complete
   206тЖТ
   207тЖТPROJECT_DIR=$(pwd)
   208тЖТPLAN_FILE="$PROJECT_DIR/.webforge/plan.md"
   209тЖТMAX_ITERATIONS=50  # Safety limit
   210тЖТ
   211тЖТfor i in $(seq 1 $MAX_ITERATIONS); do
   212тЖТ    echo "=== WebForge Loop Iteration $i ==="
   213тЖТ    
   214тЖТ    # Check if plan says COMPLETE
   215тЖТ    if [ -f "$PLAN_FILE" ]; then
   216тЖТ        if grep -q "PROJECT COMPLETE" "$PLAN_FILE"; then
   217тЖТ            echo "тЬЕ Project complete! Stopping loop."
   218тЖТ            exit 0
   219тЖТ        fi
   220тЖТ    fi
   221тЖТ    
   222тЖТ    # Launch Hermes
   223тЖТ    echo "Launching Hermes..."
   224тЖТ    opencode run "Continue the WebForge project. Read .webforge/plan.md to see what's done and what's left. Recruit agents if needed. Update the plan as you go. Say PROJECT COMPLETE only when everything is verified done."
   225тЖТ    
   226тЖТ    # Check again after Hermes finishes
   227тЖТ    if grep -q "PROJECT COMPLETE" "$PLAN_FILE" 2>/dev/null; then
   228тЖТ        echo "тЬЕ Project complete! Stopping loop."
   229тЖТ        exit 0
   230тЖТ    fi
   231тЖТ    
   232тЖТ    echo "Work remains. Relaunching Hermes in 5 seconds..."
   233тЖТ    sleep 5
   234тЖТdone
   235тЖТ
   236тЖТecho "тЪая╕П Reached max iterations ($MAX_ITERATIONS). Stopping."
   237тЖТecho "Check .webforge/plan.md for remaining work."
   238тЖТ```
   239тЖТ
   240тЖТ**How it works:**
   241тЖТ1. Script checks if plan.md says "PROJECT COMPLETE"
   242тЖТ2. If not тАФ launches Hermes via `opencode run`
   243тЖТ3. Hermes runs (35 tool calls), recruits agents, delegates, updates plan
   244тЖТ4. Hermes exits
   245тЖТ5. Script checks plan again
   246тЖТ6. If not done тАФ re-launches Hermes (fresh 35 calls)
   247тЖТ7. Repeats until complete or max iterations reached

   421тЖТ## Phase 6.7: Context Tools (Repomix + Context7 + ast-grep)
   422тЖТ
   423тЖТThese three tools solve the biggest problem with large projects like Athletica (414+ files): agents wasting all 35 tool calls just trying to understand the codebase.
   424тЖТ
   425тЖТ### Tool 1: Repomix тАФ MANDATORY for large projects
   426тЖТ
   427тЖТ**Problem:** An agent has 35 tool calls. If it reads files one by one to understand a 414-file project like Athletica, it burns 50+ calls just looking around. It never gets to do actual work.
   428тЖТ
   429тЖТ**What Repomix does:** Packs the ENTIRE project тАФ file tree + important file contents тАФ into ONE text file. The agent reads it in ONE tool call and understands the whole project instantly.
   430тЖТ
   431тЖТ**Before Repomix (wasted calls):**
   432тЖТ```
   433тЖТCall 1: read package.json
   434тЖТCall 2: read src/app/page.tsx
   435тЖТCall 3: read src/components/Header.tsx
   436тЖТCall 4: read src/components/Footer.tsx
   437тЖТ...
   438тЖТCall 35: (still reading, no work done)
   439тЖТ```
   440тЖТ
   441тЖТ**With Repomix (efficient):**
   442тЖТ```
   443тЖТCall 1: read .webforge/project-snapshot.txt (entire project packed)
   444тЖТCalls 2-35: actual work (writing code, running tests, etc.)
   445тЖТ```
   446тЖТ
   447тЖТ**Installation:**
   448тЖТ```bash
   449тЖТnpm install -g repomix
   450тЖТ```
   451тЖТ
   452тЖТ**Usage in agent system prompt:**
   453тЖТ```
   454тЖТBefore starting work, run `repomix` to pack the project into a single file.
   455тЖТRead the output file to understand the full project structure.
   456тЖТThen start working.
   457тЖТ```
   458тЖТ
   459тЖТ**For Athletica specifically:** Repomix is not optional. Without it, no agent can understand Athletica in 35 tool calls. With it, they understand it in 1 call.
   460тЖТ
   461тЖТ### Tool 2: Context7 тАФ MANDATORY (replaces library skills)
   462тЖТ
   463тЖТ**Problem:** Skill files for libraries (Next.js, Tailwind, Supabase) go outdated. Next.js updates to v16 but the skill file still describes v15. Agent follows old patterns, breaks things.
   464тЖТ
   465тЖТ**What Context7 does:** ONE MCP connection serves live, current documentation for 9,000+ libraries. When an agent needs to know "how does Next.js 16 routing work?", it asks Context7 and gets the real current docs тАФ not a stale skill file.

   477тЖТ### Tool 3: ast-grep тАФ MANDATORY for refactors
   478тЖТ
   479тЖТ**Problem:** Regular `grep` searches for TEXT, not CODE. If an agent searches for `getX` to rename it, grep finds:
   480тЖТ- `const result = getX()` тЬЕ (the function call тАФ what we want)
   481тЖТ- `// TODO: remove getX later` тЭМ (a comment тАФ false match)
   482тЖТ- `function getXName()` тЭМ (a different function тАФ false match)
   483тЖТ
   484тЖТThe agent wastes calls fixing false matches and breaks things that weren't broken.
   485тЖТ
   486тЖТ**What ast-grep does:** Understands CODE STRUCTURE. It knows the difference between a function call, a comment, and a variable name. When you say "rename every `getX` to `useX`", ast-grep only changes actual function calls.
   487тЖТ
   488тЖТ**Installation:**
   489тЖТ```bash
   490тЖТnpm install -g @ast-grep/cli
   491тЖТ```
   492тЖТ
   493тЖТ**Usage in agent system prompt:**
   494тЖТ```
   495тЖТWhen refactoring (renaming, restructuring), use ast-grep instead of grep.
   496тЖТast-grep understands code structure and avoids false matches.
   497тЖТExample: ast-grep run --pattern 'getX()' --rewrite 'useX()' --lang typescript
   498тЖТ```
   499тЖТ
   500тЖТ**For Athletica specifically:** Athletica has 414 files. When an agent needs to rename a function or restructure components, ast-grep ensures it changes the right things across all 414 files without false matches.
